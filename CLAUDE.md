## Project Overview

**komunify** is a Web3 dApp on Stellar (Soroban): one subscription, paid in a project-owned
mock USDC token, unlocks a library of PDF content published by whitelisted community
managers. Subscription revenue accrues on-chain and is attributed to content per member,
per epoch (D-009), then withdrawn by managers via `claim()`. Wallet is identity — there is
no email, no password, no user table. See `docs/PLAN.md` §1 for the exact demo script.

Packages:
- `contracts/` — Soroban (Rust) smart contracts. Cargo workspace, not an npm package.
- `packages/contract-client/` — **Generated** TypeScript bindings for both contracts (the "ABI"). Consumed by web + api.
- `packages/api/` — Hono backend API (port 3001)
- `packages/web/` — Next.js frontend (port 3000) with Freighter wallet integration
- `packages/shared/` — Shared Zod schemas, TS types, and Stellar network config/constants

## Stack

- **Blockchain**: Stellar / Soroban smart contracts (Rust, `soroban-sdk`), Stellar CLI 27+
- **Contract client**: `@stellar/stellar-sdk` v14 bindings generated by `stellar contract bindings typescript`
- **Backend**: Hono, TypeScript, Zod, Prisma (postgres)
- **Frontend**: Next.js 15 App Router, TanStack Query, Tailwind CSS, shadcn/ui, Freighter wallet (`@stellar/freighter-api`)
- **Shared**: Zod schemas, inferred TS types, Stellar network config
- **Package Manager**: bun
- **Monorepo**: bun workspaces (`packages/*`)

## Smart Contracts (Soroban)

Two contracts in the `contracts/` Cargo workspace. Signatures, types, storage keys, and the
error enum are frozen in `docs/CONTRACT_SPEC.md` — read it before touching contract code.

- **`usdc`** (`contracts/contracts/usdc/`) — mock SEP-41 token, testnet only (D-002). Full
  `TokenInterface` plus an open `faucet()` (500 USDC, 24h cooldown per address). 7 decimals,
  symbol `USDC`.
- **`komunify`** (`contracts/contracts/komunify/`) — subscriptions, content registry, read
  accounting, payouts. Attribution is per-member, not a shared pool (D-009): each
  subscription's post-fee revenue is a budget owned by that member for the epoch it was paid
  in, split only across the content that member read. No `period_secs` — the epoch is the
  billing period.

Use the **Makefile** (from repo root) for the full workflow:

```bash
make setup      # one-time: add wasm target + create/fund a testnet identity ("deployer")
make test       # cargo unit tests, both crates
make build      # compile both contracts to Wasm
make bindings   # regenerate packages/contract-client from the Wasm (both contracts) + rebuild dist
make deploy-all # deploy usdc (+init), then komunify (+init with usdc as Config.token)
make invoke   CONTRACT=komunify ARGS="subscribe --member <G...>"
make simulate CONTRACT=komunify ARGS="get_stats"
```

Then set the printed ids in `packages/web/.env.local` as `NEXT_PUBLIC_USDC_CONTRACT_ID` /
`NEXT_PUBLIC_KOMUNIFY_CONTRACT_ID`, and in `packages/api/.env` as `USDC_CONTRACT_ID` /
`KOMUNIFY_CONTRACT_ID`.

Equivalent bun script for CI: `bun contract:bindings` (wraps `make bindings`).

Prereqs: Stellar CLI (`brew install stellar-cli`), Rust, and the Wasm target
(`rustup target add wasm32v1-none`, done by `make setup`).

## Authentication

Wallet is the only identity (D-001) — no email, no password, no better-auth.

1. `POST /auth/challenge { address }` → server persists a single-use `Nonce`, returns it.
2. Browser calls Freighter `signMessage(nonce)`.
3. `POST /auth/verify { address, signature }` → server verifies the Ed25519 signature,
   marks the nonce used, sets an HTTP-only `kmf_session` cookie (JWT, `jose`, HS256).

- Required env var: `SESSION_SECRET` (32+ random bytes)
- `isManager` is simulated fresh from the contract on every request, never cached in the JWT

See `docs/API_SPEC.md` §1 for the exact request/response shapes and `docs/DECISIONS.md`
D-001 for the byte-encoding gotcha in Freighter's `signMessage`.
